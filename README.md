# Hive — a boarding-house app, and your "how websites are made" notes

This README is written as a companion to the code, not just a project blurb.
Every section explains a decision and *why* it was made that way, since
you're using this build to learn how a real website goes from a static
mock-up to something with accounts, a database, and a live deployment.

---

## 1. The big picture: what changed and why

The original `Hive` prototype was **front-end only** — plain HTML/CSS/JS with
hard-coded numbers. Every visitor saw the same "₱2,000 rent," the same fake
roommates. That's fine for a design mock-up, but it isn't a real app: nothing
you clicked was remembered, and two different people couldn't see two
different rooms.

To make it real, this project now has **two halves**:

```
Hive/
├── public/     ← the frontend: what runs in the browser (HTML/CSS/JS)
└── server/     ← the backend: a Node.js/Express API + a SQLite database
```

The frontend is still plain HTML/CSS/JS on purpose — no React, no build
step. The backend is a small Express server that stores data in a SQLite
file and exposes it as a JSON API (`/api/...`). The frontend calls that API
with `fetch()` instead of using hard-coded numbers.

**Why keep the frontend framework-free** when the reference app (Dormate)
is built in React? Because you're using this to learn the fundamentals.
React solves problems (state management, component reuse, complex UI
trees) that this app is genuinely small enough not to have yet. Every
`<div>` you can inspect in this project is *exactly* what got sent from the
server — nothing is generated by a build tool you'd have to also learn.
When the app grows past a certain size (many nested interactive components,
shared state across dozens of files), that's the natural point to reach for
a framework — and having built it by hand first will make React's
abstractions click much faster, because you'll recognize the exact problems
it's solving.

## 2. How a request flows through this app (read this first)

Understanding this one flow explains almost the whole codebase:

1. Your browser loads `public/index.html`. It pulls in `scripts/*.js`.
2. `scripts/auth.js` checks `localStorage` for a saved login token.
   - No token → show the login/signup screens (`#auth-root`).
   - Token → ask the server "who am I?" (`GET /api/auth/me`) and route you
     into the app shell (`#app`) if it's valid.
3. Once inside, `scripts/app.js` handles the sidebar: clicking "Finance"
   fetches `pages/finances.html` (a plain HTML fragment) and drops it into
   `#main-content`. This is the *exact same* fetch-a-fragment trick the
   original prototype used — we didn't change that part, it works well.
4. Each page's script (`scripts/finances.js`, `scripts/dashboard.js`,
   `scripts/hive.js`) then calls the backend, e.g.
   `GET /api/finances/overview`, and fills in the real numbers.
5. Every API call goes through `scripts/api.js`, which attaches your login
   token as an `Authorization: Bearer <token>` header. The server checks
   that token (`server/src/middleware/auth.js`) before doing anything.
6. The server's route handlers (`server/src/routes/*.js`) read/write the
   SQLite database (`server/src/db.js`) and send back JSON.

If something isn't showing up, that's the chain to trace: fragment loaded?
→ page script ran? → API call fired (check the Network tab)? → server
route hit (check the terminal running the server)? → row actually in the
database?

## 3. Why these specific technology choices

**Node.js + Express (backend)** — the most common starting point for a
JavaScript backend; if you already know JS from the frontend, you don't
need to learn a second language to write the server. Express itself is
just a thin, well-documented layer for "here's a URL, here's what to do
when it's requested."

**SQLite via Node's built-in `node:sqlite` (database)** — a real SQL
database that lives in a single file (`server/data/hive.db`), no separate
database server to install or configure. We originally used the popular
`better-sqlite3` package, but it's a *native* module — installing it means
compiling C++ on your machine, which needs a full build toolchain (Visual
Studio Build Tools on Windows, Xcode tools on Mac) and tends to break right
after a new Node version ships, before precompiled binaries catch up.
Node has shipped SQLite built in since v22.5 — zero install, zero
compiling, identical behavior on every OS and every host, and its API
(`.prepare().run()/.get()/.all()`) is close enough to `better-sqlite3`'s
that the route code looks the same either way. Requires Node **22.5+** —
see `server/package.json`'s `engines` field. The trade-off shared by any
SQLite setup: it needs a persistent disk to survive redeploys (see the
Deployment guide).

**JWT bearer tokens (auth), not cookies** — you told us you're planning a
mobile app wrapper next. A JWT is just a string the client stores and sends
back in a header; it behaves identically on a website and inside a mobile
app's HTTP client, so you won't need to rebuild auth when you wrap this.
Cookies are the more "default" web choice but come with cross-origin
headaches that don't buy you anything here.

**Hand-rolled SVG line chart, not a charting library** — see
`public/scripts/chart.js` for the full explanation in comments. Short
version: it's ~110 lines of genuinely readable code that teaches you how
charts actually work (mapping data to pixel coordinates, animating an SVG
path), instead of a black-box dependency for one chart.

## 4. What changed, feature by feature

- **Accounts & onboarding** (`public/scripts/auth.js`,
  `server/src/routes/auth.js`): sign up with email → password → name, then
  pick a role. Only **Tenant** is enabled — **Landlord** shows a disabled
  "Coming soon" option, exactly as requested. Passwords are hashed with
  bcrypt before they're stored; we never save plain-text passwords.
- **Rooms & invite codes** (`server/src/routes/rooms.js`): after picking a
  role, you either **create a room** (prompted for location, rent, and
  notes — location and notes are optional) which immediately generates a
  shareable code like `HIVE-6UQ8ZR`, or **join with a code**, which doesn't
  add you right away — it posts a notification.
- **Join requests as Announce-tab notifications**
  (`public/scripts/hive.js` → `renderAnnouncementCard`): when someone
  tries to join, a special card appears on the Announce tab of every
  *current* member with Accept/Decline buttons. Whoever responds first
  resolves it for the whole room.
- **Dashboard status** (`public/pages/dashboard.html`,
  `public/scripts/dashboard.js`): the old "scan to update status" button is
  gone. Tapping Home / In Room / Outside sets it directly.
- **Finances**:
  - The **"You Owe"** card and per-item owed amounts are removed. Shared
    expenses now just track a total and a Settled/Unsettled toggle.
  - Bills (Electric/Water) have a real **"Mark Paid"** button.
  - The Overview and Bills bar charts are now an **animated line chart**
    with a soft gradient fill, matching the Dormate reference's motion.
- **Poll** (renamed from "Meal Poll"): anyone can create a poll with 2+
  options and an optional deadline. Polls close automatically once the
  deadline passes.
- **Schedule → Edit Your Schedule**: you manage your own list of weekly
  sessions (day + time + title) and can mark each one **visible** or not.
  Roommates viewing your schedule still see *that* you're busy at that
  time — just not what for, if you hid it.
- **Settings**: view your email/role, change your display name, change
  your profile photo (stored as a base64 image — fine at this scale; for a
  large user base you'd move to real file storage, e.g. S3).

## 5. Project structure

```
Hive/
├── public/                  # served as static files, this is "the website"
│   ├── index.html           # shell: auth screens + app shell + settings modal
│   ├── pages/                # HTML fragments loaded into #main-content
│   │   ├── dashboard.html
│   │   ├── finances.html
│   │   └── hive.html
│   ├── scripts/
│   │   ├── api.js           # fetch wrapper, attaches the login token
│   │   ├── auth.js          # login/signup/role/room onboarding screens
│   │   ├── app.js           # sidebar nav + page-fragment loader
│   │   ├── dashboard.js
│   │   ├── finances.js
│   │   ├── hive.js          # announce / bulletin / poll / schedule
│   │   ├── chart.js         # the animated SVG line chart
│   │   └── settings.js
│   ├── style/style.css
│   └── images/
└── server/
    ├── package.json
    ├── .env.example          # copy to .env
    ├── data/                 # hive.db lives here (gitignored)
    └── src/
        ├── index.js           # Express app entry point
        ├── db.js              # SQLite connection + schema
        ├── middleware/auth.js # JWT verification
        ├── utils/roomCode.js
        └── routes/            # one file per feature area
```

## 6. Run it locally

You'll need **Node.js 22.5 or newer** (check with `node --version`) — that's
the version SQLite support shipped in. Get it from nodejs.org if needed.

```bash
cd Hive/server
cp .env.example .env      # then edit JWT_SECRET to any random string
npm install
npm start
```

You'll see a one-time `ExperimentalWarning: SQLite is an experimental
feature` printed on startup — that's expected and harmless, it just means
Node's own SQLite API could still change in a future Node release. Nothing
to fix.

Open **http://localhost:8000** — that's it. One process serves both the
API (`/api/...`) and the website itself, so there's nothing else to run or
configure for local development.

Create a first account through the sign-up flow, create a room, and note
the invite code it gives you. Open the site again in a private/incognito
window (or another browser) to sign up as a second "roommate" and join
with that code — then accept the request from the first account's
Announce tab.

If you ever want a clean slate, stop the server and delete
`server/data/hive.db*`.

## 7. Where to go next (learning suggestions)

- Open the Network tab in your browser's dev tools while you click around —
  watch the actual `fetch()` calls and JSON responses happen.
- Open `server/data/hive.db` with a SQLite browser (e.g. "DB Browser for
  SQLite") while the app is running, to see your data as literal rows.
- Try adding a new small feature end-to-end yourself (e.g. a "delete
  announcement" button) — you'll touch a route, the SQL, and the frontend
  fetch call, which is the full stack in miniature.

See **DEPLOYMENT.md** in this same folder for how to put this online, and
notes on what changes once you wrap it as a mobile app.
